# エージェントガイド

## ドキュメントの同期

- `README.md` と `README-ja.md` は意味を揃えて保つ。
- `AGENTS.md` と `AGENTS-ja.md` は意味を揃えて保つ。
- 対になる文書の片方を更新する場合、ユーザーが一時的な不一致を明示的に求めていない限り、同じ変更の中でもう片方も更新する。
- `*-ja.md` は逐語訳ではなく、自然な日本語で書く。

## 参照元の分担

- 公開する挙動、CLI の使い方、例、ユーザー向けのスコープは `README.md` と `README-ja.md` を基準にする。
- この文書には、エージェント向けの作業ルール、内部設計の境界、リポジトリ保守メモ、README に置かない実装指針を書く。
- 公開挙動を変える場合は、同じ変更の中で README の対になる文書も更新する。
- 内部向けの指針を変える場合は、同じ変更の中で AGENTS の対になる文書も更新する。

## README 用のメディア

- README のデモ GIF は `assets/demo.tape` から生成する。
- デモを更新するときは、GIF を直接編集せず `assets/demo.gif` を再生成する。
- デモ生成には `vhs`、`ttyd`、`ffmpeg`、`tree`、そして `PATH` 上で実行できる `pouch` バイナリが必要である。
- 標準的な再生成手順は次のとおり。
  - リポジトリの外に、空の作業ディレクトリを作成する。
  - `vhs`、`ttyd`、`ffmpeg`、`tree`、`pouch` が `PATH` 上で使えることを確認する。
  - その作業ディレクトリで `vhs /absolute/path/to/repo/assets/demo.tape` を実行する。
  - 生成された `demo.gif` をリポジトリ内の `assets/demo.gif` にコピーする。
  - 生成された `assets/demo.gif` が README のコマンド例と実際の CLI 挙動に合っていることを確認する。
  - 一時作業ディレクトリを削除する前に、ユーザーへ確認する。

## リポジトリの境界

- `pouch` は CLI ツールである。
- ルートの Go パッケージを、外部向けに保証するライブラリ API とはみなさない。
- プロジェクトの焦点は、パスから直感的にファイルやディレクトリを作成することに置く。
- 汎用スキャフォールディングツールへは広げない。

## 内部設計の境界

- パス判定は決定的で、副作用を持たないように保つ。
- 単一パスの作成を、振る舞いの基本単位にする。
- 複数パスの調停処理は小さく保つ。入力順を守り、最初のエラーで止まり、失敗前までの成功結果を返したうえでエラーも返す。
- 自動判定を賢くしすぎるより、明示モードで上書きできる設計を優先する。
- 役割分担は次のまま保つ。
  - `internal/cli` は引数解析と CLI の検証を担当する。
  - `internal/pouch/detect.go` は判定ロジックだけを担当する。
  - `internal/pouch/create.go` はファイルシステム上の挙動を担当する。
  - `internal/pouch/pouch.go` は複数パス処理の調停を担当する。
- 必要性が明確になるまでは、これ以上パッケージを分割しない。

## 実装メモ

### パス処理

- 標準ライブラリの `path/filepath` を使う。
- `mkdir` や `touch` などの OS コマンドは呼ばない。
- 独自の正規化より、標準ライブラリの挙動を優先する。

### ファイル作成

- ファイルを作成する前に、親ディレクトリを確実に作る。
- `os.OpenFile` を create 可能なフラグ付きで使う。
- ファイルはすぐ閉じる。
- 公開仕様が変わらない限り、既存ファイルは変更しない。

### 権限

- コード上では明示的な既定値を使う。
  - ディレクトリ: `0o755`
  - ファイル: `0o644`
- 実際の有効権限は `umask` の影響を受ける。

### エラー

- エラーにはパス文脈を付けて包む。
- 独自の複雑なエラー階層より、平明で直接的なメッセージを優先する。
- エラーメッセージの言い回しは README にある CLI の挙動と矛盾しないように保つ。

## テスト指針

- 標準のテストコマンドは `go test ./...` とする。
- リポジトリルート自体に build 対象の Go パッケージがあるとは限らないものとして扱う。
- 見通しがよくなる箇所ではテーブル駆動テストを使う。
- ファイルシステムの隔離には `t.TempDir()` を使う。
- 実装の細部ではなく、観測可能な挙動を中心にカバーする。
- テスト範囲は README にある現在の公開仕様に合わせて保つ。特に次は外さない。
  - 自動判定
  - `--file`、`--dir`、`--mode` による明示モード上書き
  - `--mode` とモード指定ショートカットの競合
  - 親ディレクトリ作成
  - dry-run 挙動
  - 既存パスに対する strict 挙動
  - 後置きフラグの解析
  - パス風の引数に対する `--` の扱い

## レビュー観点

- コードは README に書かれた公開挙動と一致しているか。
- CLI 中心の構成は、その背後の実装より小さく単純なままか。
- 曖昧なケースを、新しいヒューリスティックで隠していないか。
- テストは観測可能な挙動を見ているか。
- README の例を変えた場合、デモテープも追従しているか。

## 変更の規律

- ヒューリスティックを勝手に追加しない。
- 公開挙動を広げるときは、README もあわせて更新する。
- README の例やデモに見える挙動を変えた場合は、`assets/demo.tape` を更新して `assets/demo.gif` を再生成する。
- 小さくて予測しやすいパス作成ツールという中核の性格を保つ。
